02Что уже есть на рынке

Готовые кирпичи под наш кейс

Git-first registry

Реестры схем (напр. Apicurio 3.3, июнь 2026) с GitOps-хранилищем: git — источник правды, реестр read-only отдаёт по HTTP, запись только через PR.

GitOps + registry + MCP

Целостная петля: агент забирает точные схемы из реестра через MCP и генерирует код под них. Нулевой дрифт, потому что данные живые.

Drift-guardrails в PR

Инструменты ловят ломающие изменения контракта в pull request — до того как они каскадом сломают агентные workflow. Актуально позже.

Вывод: ничего кастомного строить с нуля не нужно. Общий git-репо в роли реестра + тонкий слой доступа для агентов покрывает всё на нашем масштабе.

03Главный принцип

Git — единственный источник правды.
Агенты читают из него напрямую.

Никакой синхронизации, никаких копий, никакого лага. И людей нет в петле.

Контракт, который пересекает границу владельца, не может жить внутри приватного репозитория одного из участников — иначе это не источник правды, а чья-то приватная копия. Место контракта определяет владение сервисом, а не то, кто первый его записал.

04Топология

Общий приватный репо как нейтральная земля

сплошная — пишет свою секцию (git PR) пунктир — читает через MCP (read-only)
05Ключевое различие

Два разных артефакта. Оба называются «контракт».

provides

«Вот что мой сервис отдаёт»

Владелец — тот, чей сервис реально выдаёт эти данные / API. Есть у продюсера данных, у продюсера контента. У Axon почти пуст — мы конечная точка.

consumes

«Вот что мне нужно сверху»

Тоже полноценный производимый контракт, но другой природы — это требования. Именно его производит Axon активнее всех.

Разные роли — одно хранилище. Каждый владелец пишет только в свою папку; потребители пишут ожидания в consumes/<чужой-сервис>/.

06Роль Axon

Axon производит требования, а не поставку

Ты — финальный потребитель, и вся цепочка ориентируется на твои требования. Спрос идёт сверху вниз: ты говоришь «мне нужны данные вот в таком виде» → продюсер контента подстраивается → транслирует ниже продюсеру данных.

  • Axon — источник спроса, а не источник поставки. Оба — контракты, оба публикуются, оба читаются всеми.
  • То, что Axon производит контракты, не значит, что они живут в Axon. Они в общем репо, в папке axon/, которую пишешь только ты.
  • Приватный монорепо Axon остаётся полностью закрытым — наружу выходит только секция контрактов, которую ты и так намеренно публикуешь.
07Механика каскада требований

Ты меняешь требование — оно докатывается по цепочке

Axonфинальный потребитель
axon/consumes/content-producer/ ✎ Меняешь требование: «теперь мне нужно поле publishedAt в ISO-8601». webhook → чат: затронут продюсер контента
Продюсер контентасреднее звено
content-producer/consumes/data-producer/ ✎ Агент видит новое требование сверху. Чтобы его выполнить, ему самому нужны сырые данные с датой — он обновляет своё требование вниз. webhook → чат: затронут продюсер данных
Продюсер данныхисточник · хаб
data-producer/provides/ ✎ Агент видит требование и добавляет поле в свой provides. Каскад дошёл до источника — цепочка снова согласована. другие потребители хаба тоже видят диф
Каждый шаг — обычный git-коммит в свою секцию. Никто не пишет в чужую.
08Модель доступа

Приватный наружу, открытый внутри

Видимость наружу

Репо контрактов всегда private. Приватный монорепо Axon не расшаривается вообще — это остаётся неизменным.

Доступ внутри

Все участники — read + write на общий репо. «Приватный» и «доступ всей команды» — одно состояние, не противоречие.

Запись ≠ чтение

Пишут контракты только через git-коммит (PR владельца). MCP-сервер — только на чтение. Git пишем — реестр отдаёт.

Как подключается Axon: локальный MCP-сервер на Go читает origin/main общего репо после git fetch (не рабочее дерево — иначе агент прочитает свои незакоммиченные правки). Ноль копий контрактов внутри Axon.

09Порядок внедрения

По нарастанию — без стройки платформы

Уровень 0
Структура contracts/Секции по владению сервисом с папками provides / consumes / drafts. Draft — явный статус, а не отсутствие файла. Час работы.
Уровень 1
Уведомления об измененияхGitHub webhook на contracts/** → в командный чат с дифом. Событийно, не крон — без лага.
Уровень 2
MCP-сервер над контрактамиGo-сервер отдаёт контракты как tools: get_contract, list_drafts, whats_changed_since. Главный рычаг — агент спрашивает, а не парсит файлы.
Уровень 3
Валидация в CISchema-first проверка дрифта в PR. Добавляем, когда drafts начинают стабилизироваться в provides. Раньше — преждевременно.
10Честная оговорка

Схема работает только с теми, кто согласился играть

Партнёры в деле

Пишут свою секцию напрямую

Вы + кооперирующиеся команды. Каждый ведёт свою папку в общем репо. Полная автоматизация возможна.

Неподконтрольный вендор

Зеркалим read-only

По-настоящему внешний сервис публикует спеку по-своему. Мы зеркалим её в общий репо как «вендоренный» контракт и сами держим зеркало актуальным. Это кэш чужого источника, не источник.

Это важно проговорить с партнёрами заранее: автоматический обмен требует, чтобы каждая сторона писала свою секцию. Там, где стороны нет за столом, ответственность за актуальность зеркала берёт кто-то у нас.

11Итог

Одной фразой

Один общий приватный репо контрактов как нейтральная земля: каждый владелец пишет свою секцию, читают все через MCP только на чтение, приватный код наружу не выходит.

  • Сегодня: структура contracts/ с provides / consumes / drafts.
  • На неделе: webhook изменений в чат.
  • Дальше: MCP-сервер на Go, когда агенты начнут спотыкаться об файлы.

Не строим кастомный роутер пушей/пуллов. Не трогаем A2A / agent-cards. Не заводим крон. Git + webhook + MCP покрывают весь граф.
