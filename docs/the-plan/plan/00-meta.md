# §0 Meta & Glossary

> Part of the a2ahub architecture plan. Index: [00-STRUCTURE.md](00-STRUCTURE.md).

## 0.1 How to read this document set

- **Audience:** AI implementer agents first, humans second. The set assumes
  zero prior context: every domain term is defined in §0.4; every participant
  in §0.5; every cross-reference is by stable ID.
- **Normative language:** the key words MUST, MUST NOT, SHOULD, SHOULD NOT,
  MAY are to be interpreted as in RFC 2119. Non-normative text is prefixed
  `NOTE:` or `RATIONALE:`.
- **No implementation code.** This set specifies *what* declaratively —
  schemas as field tables, lifecycles as transition tables, APIs as operation
  tables. Filled-in example *documents* (Appendix B) are data, not code, and
  are normative as golden fixtures.
- **Precedence:** if sections conflict, the Decision Log (§17) wins, then the
  domain model (§3), then other sections. Conflicts found by implementers
  MUST be reported back as plan defects, not silently resolved.

## 0.4 Glossary

Terms are lower-case in running text; definitions here are normative.

| Term | Definition |
|---|---|
| **actor** | Any authenticated identity that reads or writes exchange artifacts: a human or an agent, always attributed to a system and an org (§2). |
| **agent** | An AI worker (Claude Code, Codex, CI job, …) acting for a system. Agent vendor/model is metadata, never protocol. |
| **artifact** | Any typed, schema-validated document stored in a space: one of the object types of §3. |
| **binary / `a2a`** | The single distributable Go executable: CLI + local MCP server + validator + statusline provider + local HTML generator (§7). |
| **cascade** | The propagation of a requirement down a dependency chain, each hop being an ordinary commit into the author's own section (§1.3). |
| **circle** | Informal term for the set of participants of one space. |
| **contract (provides)** | A versioned interface description a party publishes and others may implement against (§3). |
| **digest** | A content hash of an artifact body, used for immutable references and drift detection. |
| **durable artifact** | Any artifact whose SSOT is git (all 8 object types + their lifecycle events), per D-001. |
| **envelope** | The type-independent metadata layer every artifact carries (frontmatter): identity, addressing, lifecycle, provenance, references (§5). |
| **ephemeral state** | State owned by the hub and rebuildable or discardable: indexes, watermarks, subscriptions, presence, notification delivery status (D-001). |
| **exchange** | One request-shaped artifact plus its lifecycle events and response(s), as a unit of tracking. |
| **exchange space / space** | One git repository shared by a circle: the neutral ground and SSOT for their exchanges (D-003). |
| **federation** | A system participating in multiple spaces; the binary/hub aggregate all connected spaces uniformly (D-003). |
| **handoff** | A typed artifact transferring implemented and tested work to agents in another system, self-sufficient for a zero-context consumer (§16). |
| **hub** | The optional Go service on the operator's VPS: index, notification fan-out, dashboard. Owns ephemeral state only (§6). |
| **lifecycle event** | An append-only, committed record of a state transition of an exchange (submitted, acknowledged, declined, …) (§3). |
| **operator** | The human who administers the hub and owns the org hosting spaces. Currently Yura. |
| **participant** | A system that is a member of a space. |
| **producer-outbox** | The pre-a2ahub folder of relayed exchange files in the axon repo; migration seed for v1 (R-022). |
| **requirement (consumes)** | A published demand on another party's contract: "this is what I need" (§3). |
| **section** | The subtree of a space owned and writable by exactly one system (enforced by CODEOWNERS + branch protection). |
| **spool** | A project-local, non-normative projection of a system's exchanges (inbox/outbox views) generated by the binary; never a second SSOT. |
| **SSOT** | Single source of truth. Exactly one writable authority exists per object class (D-001). |
| **statusline provider** | The binary's mode that emits a one-line summary of pending inbound items for embedding into an agent harness statusline (Claude Code first) (§7). |
| **system** | A software project participating in exchanges (e.g. axon, seomatrix). The addressing unit: artifacts are addressed system→system. |
| **template** | The canonical skeleton document for an object type, shipped with the binary, from which agents draft artifacts (R-015). |
| **vendored mirror** | A read-only copy of a non-participating vendor's spec kept current by a designated participant; a cache, not a source (§4). |
| **watermark / delivery cursor** | Hub-internal marker of the last delivered event per notification channel (6.5); not a public API concept in v1. |

## 0.5 Canonical participant names (v1)

| System ID | Org | Role in chain | Agents |
|---|---|---|---|
| `axon` | Yura | Web platform (getvisa): backend + frontend, SEO/GEO delivery; final consumer of content/data; source of demand | Claude Code, Codex |
| `seomatrix` | Misha | Content factory: verification, facts, content production; middle of chain | Claude |
| `sot` *(planned)* | partner | Source-of-truth startup: visa data | TBD |

System IDs are lowercase kebab, globally unique within all spaces they join,
and appear in every artifact's addressing metadata. The v1 space is the
getvisa chain circle: `axon` + `seomatrix` (D-007), `sot` joins later via the
§9 onboarding runbook.
